# 会议室预约管理系统开发文档

# 1. 文档概述

## 1.1 开发背景

为解决办公楼会议室传统预约方式效率低下、易出现时间冲突、管理不规范等问题，基于Python语言开发会议室预约管理系统，采用RPC（远程过程调用）架构实现客户端与服务器端互联，完成核心功能开发（预约会议室、按ID查询预约、按组织者查询预约、取消预约），适配办公楼实际场景（6层，每层3个10人会议室、2个5人会议室），本文档为系统开发的唯一技术依据，明确开发规范、实现逻辑及技术细节。

## 1.2 开发目标

1.  采用Python语言开发，基于RPC架构实现客户端与服务器端互联，确保前后端通信稳定、无数据传输异常；

2.  严格实现4个核心功能，确保业务逻辑正确（时间冲突校验、数据校验等），无功能遗漏、无逻辑bug；

3.  客户端采用tkinter库开发模拟APP风格GUI界面，实现登录、点击操作、返回、退出等交互逻辑，操作流畅、反馈及时；

4.  服务器端采用内存存储预约数据，确保数据存储可靠，支持多客户端并发请求，满足办公楼日常使用需求；

5.  代码规范、注释清晰，便于后期维护和调试，无冗余代码，符合Python开发规范。

## 1.3 文档范围

本文档覆盖系统开发全流程，包含技术栈选型、系统架构实现、核心模块开发、数据结构定义、代码实现、交互逻辑开发、测试规范、部署说明等内容，聚焦开发技术细节，不包含产品需求分析类内容，为开发人员提供明确的技术指导。

## 1.4 开发环境与技术栈

### 1.4.1 开发环境

1.  编程语言：Python 3.7及以上版本；

2.  操作系统：Windows 10/11、Linux（CentOS/Ubuntu）、MacOS均可；

3.  开发工具：PyCharm、VS Code等Python开发工具；

4.  依赖环境：无额外第三方依赖，基于Python原生库开发。

### 1.4.2 核心技术栈

1.  后端技术：Python原生socket（实现RPC通信）、多线程（处理并发请求）、内存列表（数据存储）；

2.  前端技术：Python tkinter库（开发模拟APP风格GUI界面）；

3.  数据序列化：JSON格式（实现客户端与服务器端数据传输）；

4.  通信协议：TCP协议（基于socket实现可靠传输）。

# 2. 系统架构设计与实现

## 2.1 整体架构设计

系统采用客户端-服务器（C/S）架构，基于Python原生RPC机制实现前后端互联，核心分为客户端（GUI交互层）、服务器端（业务逻辑+数据存储层），架构清晰、分工明确，无多余模块设计，具体架构如下：

客户端（Client）：负责GUI界面展示、用户交互（登录、核心功能操作、返回、退出）、RPC请求发送、服务器端响应接收与展示；

服务器端（Server）：负责RPC服务启动、客户端请求监听与处理、业务逻辑校验（时间冲突、数据合法性）、预约数据内存存储、核心接口实现；

RPC通信层：基于Python原生socket实现，负责客户端与服务器端的数据序列化/反序列化、请求传输、响应反馈，屏蔽网络通信细节。

## 2.2 服务器端架构实现

### 2.2.1 核心模块划分

服务器端分为3个核心模块，模块间低耦合、高内聚，具体如下：

1.  服务启动模块：负责启动RPC服务、绑定端口、监听客户端连接、开启多线程处理并发请求；

2.  业务逻辑模块：负责实现4个核心接口的业务逻辑，包含数据校验、时间冲突校验等；

3.  数据存储模块：负责预约数据的内存存储（Python列表）、数据新增、删除、查询等操作，维护自增预约ID。

### 2.2.2 服务启动实现逻辑

1.  采用socket创建TCP服务端，绑定IP（127.0.0.1）和端口（8888），设置端口复用，避免端口占用冲突；

2.  调用socket.listen()方法监听客户端连接，设置最大连接数为5，支持5个客户端并发请求；

3.  采用多线程（threading.Thread）处理每个客户端连接，每个客户端请求对应一个独立线程，避免阻塞主服务；

4.  线程内接收客户端传入的JSON格式请求数据，解析请求类型，调用对应业务逻辑方法，处理完成后返回JSON格式响应数据。

## 2.3 客户端架构实现

### 2.3.1 核心模块划分

客户端分为3个核心模块，基于tkinter库开发，模拟APP图形界面风格：

1.  GUI界面模块：负责登录页面、主界面、各功能页面（预约、查询、取消）的布局与组件实现；

2.  交互逻辑模块：负责按钮点击事件、输入校验、页面跳转（登录→主界面、功能页面→主界面、退出→登录页面）、操作反馈展示；

3.  RPC请求模块：负责与服务器端建立连接，发送JSON格式请求数据，接收服务器端响应，解析并反馈给用户。

### 2.3.2 GUI界面实现规范

1.  界面布局：采用tkinter的grid或pack布局，模拟APP风格，组件排列整齐，按钮、输入框大小适配，视觉简洁；

2.  组件要求：包含标签（Label）、输入框（Entry）、按钮（Button）、列表框（Listbox）、弹窗（messagebox）等组件，满足交互需求；

3.  页面切换：登录页面完成登录后跳转至主界面，各功能页面点击“返回”按钮跳转至主界面，点击“退出”按钮返回登录页面；

4.  反馈展示：操作结果（成功/失败）通过tkinter.messagebox弹窗展示，输入错误时实时提示。

## 2.4 RPC通信实现

1.  数据传输格式：采用JSON格式，客户端发送请求时，包含“action”字段（指定操作类型：登录、book、queryById、queryByOrganizer、cancel）和对应参数；

2.  序列化与反序列化：客户端发送请求前，将Python字典转换为JSON字符串（json.dumps()）；服务器端接收后，将JSON字符串转换为Python字典（json.loads()），处理完成后，将响应数据转换为JSON字符串返回；

3.  通信流程：客户端创建socket连接→发送请求数据→服务器端接收并处理→服务器端返回响应数据→客户端接收并解析→展示操作结果；

4.  异常处理：客户端连接服务器失败、网络中断时，弹出提示；服务器端接收非法JSON数据时，返回错误响应，避免服务崩溃。

# 3. 核心数据结构定义

系统核心数据结构为预约对象（Meeting），用于存储预约相关信息，作为客户端与服务器端数据传输的核心载体，字段定义如下（Python字典格式）：

|字段名|字段类型（Python）|字段说明|约束条件与实现说明|
|---|---|---|---|
|meetingId（预约ID）|int|预约记录的唯一标识，用于查询、取消预约|服务器端维护全局自增变量next_id，初始值为1，每次新增预约自动+1，确保全局唯一|
|organizerName（组织者姓名）|str|会议发起人的姓名，用于按组织者查询预约|非空，客户端输入时校验，服务器端再次校验，不可为空字符串|
|roomName（会议室名称）|str|办公楼内被预约的会议室名称|符合命名规范：6层，每层3个10人会议室（X层10人-1/2/3）、2个5人会议室（X层5人-1/2），X为1-6，客户端输入时校验格式|
|topic（会议主题）|str|本次会议的主题或用途|非空，客户端输入时校验，服务器端再次校验，不可为空字符串|
|startTime（开始时间）|str|会议预约的开始时间|格式为“yyyy-MM-dd HH:mm”，早于结束时间，不可为过去时间，客户端和服务器端均需校验格式与逻辑|
|endTime（结束时间）|str|会议预约的结束时间|格式为“yyyy-MM-dd HH:mm”，晚于开始时间，客户端和服务器端均需校验格式与逻辑|
|attendeeCount（参与人数）|int|本次会议的参会人数|大于0的整数，客户端输入时校验（需为整数且大于0），服务器端再次校验|
### 3.1 辅助数据结构

1.  服务器端内存存储：采用Python列表meeting_list，用于存储所有预约对象（Meeting），列表元素为上述Meeting字典；

2.  客户端请求数据结构：Python字典，包含“action”（操作类型）和对应参数，示例如下：

预约请求：{"action": "book", "organizerName": "张三", "roomName": "1层10人-1", "topic": "部门会议", "startTime": "2026-04-15 14:00", "endTime": "2026-04-15 15:00", "attendeeCount": 8}

3.  服务器端响应数据结构：Python字典，包含“success”（操作结果）、“msg”（提示信息）、“data”（返回数据，如预约ID、预约列表）。

# 4. 核心模块开发实现

## 4.1 服务器端核心模块开发

### 4.1.1 服务启动模块（server.py）

核心功能：启动RPC服务，监听客户端连接，开启多线程处理并发请求，代码实现逻辑如下：

1.  导入所需模块：socket、threading、json、datetime；

2.  初始化全局变量：meeting_list（预约数据存储列表）、next_id（自增预约ID，初始值1）；

3.  定义handle_client(conn)函数：处理单个客户端请求，接收请求数据、解析请求类型、调用对应业务方法、返回响应数据；

4.  定义start_server()函数：创建socket服务端、绑定端口、监听连接，循环接收客户端连接，开启线程处理；

5.  主程序入口：调用start_server()启动服务，打印服务启动提示。

### 4.1.2 业务逻辑模块（server.py）

实现4个核心接口的业务逻辑，每个接口对应一个处理函数，具体实现如下：

#### 1. 预约会议室（bookMeeting）

函数功能：接收客户端预约请求，校验数据合法性、时间冲突，新增预约记录，返回预约结果；

实现步骤：

① 解析客户端传入的预约参数（organizerName、roomName、topic、startTime、endTime、attendeeCount）；

② 数据合法性校验：校验所有字段非空，attendeeCount为大于0的整数，startTime、endTime格式正确且startTime < endTime，startTime不早于当前时间；

③ 时间冲突校验：调用is_conflict()函数，判断目标会议室在预约时间段内是否已有预约（冲突判定：新预约[s1,e1)与已有预约[s2,e2)满足e1 > s2 且 s1 < e2）；

④ 新增预约：校验通过后，为预约分配next_id，将预约对象添加到meeting_list，next_id自增1；

⑤ 返回响应：返回{"success": True, "msg": "预约成功", "meetingId": 新增预约ID}，校验失败则返回{"success": False, "msg": "失败原因"}。

#### 2. 按ID查询预约（queryByID）

函数功能：接收客户端传入的预约ID，查询对应的预约记录，返回查询结果；

实现步骤：

① 解析客户端传入的meetingId（整数类型）；

② 遍历meeting_list，匹配meetingId与列表中预约对象的meetingId；

③ 找到匹配记录：返回{"success": True, "data": 预约对象字典}；

④ 未找到匹配记录：返回{"success": False, "msg": "无此预约ID"}。

#### 3. 按组织者查询预约（queryByOrganizer）

函数功能：接收客户端传入的组织者姓名，查询该组织者发起的所有预约记录，返回查询结果；

实现步骤：

① 解析客户端传入的organizerName；

② 遍历meeting_list，筛选出organizerName与输入一致的所有预约对象；

③ 返回响应：{"success": True, "data": 预约对象列表}（无匹配记录时，data为空列表）。

#### 4. 取消预约（cancelMeeting）

函数功能：接收客户端传入的预约ID，删除对应的预约记录，返回取消结果；

实现步骤：

① 解析客户端传入的meetingId（整数类型）；

② 遍历meeting_list，匹配meetingId与列表中预约对象的meetingId；

③ 找到匹配记录：删除该预约对象，返回{"success": True, "msg": "取消成功"}；

④ 未找到匹配记录：返回{"success": False, "msg": "无此预约ID，取消失败"}。

### 4.1.3 数据存储模块（server.py）

核心功能：维护预约数据的内存存储，提供数据新增、删除、查询操作，实现如下：

1.  数据存储载体：Python列表meeting_list，初始为空列表；

2.  数据新增：预约成功时，将预约对象字典添加到meeting_list；

3.  数据删除：取消预约成功时，从meeting_list中删除对应预约对象；

4.  数据查询：查询功能（queryByID、queryByOrganizer）通过遍历meeting_list实现；

5.  数据生命周期：服务器端启动后，meeting_list初始化为空；服务器端关闭后，内存数据丢失，重启后重新初始化。

## 4.2 客户端核心模块开发

### 4.2.1 GUI界面模块（client.py）

基于tkinter库开发，实现3个核心页面：登录页面、主界面、功能操作页面，具体布局与组件实现如下：

#### 1. 登录页面

组件布局：

① 标题标签（Label）：“会议室预约管理系统”；

② 用户名输入框（Entry）：提示“请输入用户名”；

③ 登录按钮（Button）：点击触发登录逻辑，验证通过后跳转至主界面；

④ 提示标签（Label）：用于显示登录错误提示（如“用户名不能为空”）；

实现逻辑：登录页面为客户端启动后的默认页面，用户名非空即可登录（无需复杂权限校验），登录成功后销毁登录页面，创建并显示主界面。

#### 2. 主界面

组件布局：

① 标题标签（Label）：“会议室预约系统 - 主界面”；

② 功能按钮（4个）：“预约会议室”“按ID查询”“按组织者查询”“取消预约”，点击对应按钮跳转至对应功能页面；

③ 退出按钮（Button）：点击弹出确认弹窗，确认后销毁主界面，返回登录页面；

实现逻辑：主界面为核心功能入口，所有功能页面均从主界面跳转，点击“返回”按钮可回到主界面。

#### 3. 功能操作页面

每个功能页面均包含“返回”按钮（跳转至主界面）、“退出”按钮（返回登录页面），具体组件如下：

① 预约会议室页面：6个输入框（组织者姓名、会议室名称、会议主题、开始时间、结束时间、参与人数）、“预约”按钮、提示标签；

② 按ID查询页面：1个输入框（预约ID）、“查询”按钮、列表框（展示查询结果）、提示标签；

③ 按组织者查询页面：1个输入框（组织者姓名）、“查询”按钮、列表框（展示查询结果）、提示标签；

④ 取消预约页面：1个输入框（预约ID）、“取消”按钮、提示标签。

### 4.2.2 交互逻辑模块（client.py）

核心功能：处理按钮点击事件、输入校验、页面跳转、操作反馈，具体实现如下：

1.  登录逻辑：点击登录按钮，校验用户名非空，非空则跳转至主界面，否则弹出提示；

2.  页面跳转逻辑：登录→主界面、主界面→各功能页面、功能页面→主界面、任意页面→登录页面（退出）；

3.  输入校验逻辑：对各功能页面的输入框进行校验（非空、格式正确，如时间格式、整数校验），输入错误时弹出提示；

4.  操作反馈逻辑：操作完成后（预约、查询、取消），通过messagebox弹窗展示结果（成功/失败）。

### 4.2.3 RPC请求模块（client.py）

核心功能：与服务器端建立连接，发送请求、接收响应，实现如下：

1.  定义request(data)函数：接收请求数据（Python字典），转换为JSON字符串，发送至服务器端（IP：127.0.0.1，端口：8888）；

2.  接收服务器端响应数据，转换为Python字典，返回给调用者；

3.  异常处理：连接失败、网络中断时，弹出“连接服务器失败，请启动server.py”提示；

4.  调用时机：用户点击“预约”“查询”“取消”按钮后，调用该函数发送对应请求，接收响应后展示结果。

# 5. 交互逻辑开发细节

## 5.1 整体交互流程

1.  客户端启动→展示登录页面→用户输入用户名→点击登录→跳转至主界面；

2.  主界面→点击对应功能按钮→跳转至功能操作页面；

3.  功能操作页面→输入相关信息→点击操作按钮→发送RPC请求→接收响应→展示操作结果；

4.  功能操作页面→点击“返回”→跳转至主界面；

5.  任意页面→点击“退出”→弹出确认弹窗→确认后返回登录页面，取消则留在当前页面。

## 5.2 关键交互实现

1.  登录交互：用户名输入框绑定“回车”事件，支持回车登录；登录按钮点击后，校验通过则销毁登录页面，创建主界面；

2.  返回交互：各功能页面的“返回”按钮点击后，销毁当前功能页面，显示主界面；

3.  退出交互：点击“退出”按钮，弹出messagebox.askyesno()确认弹窗，点击“是”则销毁当前页面，显示登录页面；点击“否”则关闭弹窗，留在当前页面；

4.  操作反馈：所有操作结果（成功/失败）通过messagebox.showinfo()或messagebox.showerror()弹窗展示，弹窗关闭后，清空输入框内容（查询、取消功能除外）。

# 6. 代码规范

## 6.1 命名规范

1.  文件名：全部小写，采用下划线命名法，如server.py、client.py；

2.  变量名：全部小写，采用下划线命名法，如meeting_list、next_id、organizer_name；

3.  函数名：全部小写，采用下划线命名法，如handle_client、start_server、is_conflict；

4.  类名（若有）：采用驼峰命名法，如MeetingRoomService；

5.  常量名：全部大写，采用下划线命名法，如SERVER_IP、SERVER_PORT。

## 6.2 注释规范

1.  头部注释：每个文件顶部添加注释，说明文件功能、开发日期、作者（可选）；

2.  函数注释：每个函数顶部添加注释，说明函数功能、参数含义、返回值含义；

3.  代码注释：关键代码行（如校验逻辑、冲突判断、数据处理）添加单行注释，说明代码功能；

4.  注释清晰、简洁，避免冗余，不注释显而易见的代码。

## 6.3 代码格式规范

1.  缩进：采用4个空格缩进，不使用Tab缩进；

2.  空行：函数之间、模块之间添加1-2个空行，提高代码可读性；

3.  语句规范：每行仅写一条语句，避免一行多句；

4.  异常处理：所有可能出现异常的代码（如socket连接、JSON解析、时间格式转换）均需添加try-except捕获异常，给出明确提示，避免程序崩溃。

# 8. 部署说明

## 8.1 部署环境准备

1.  安装Python 3.7及以上版本，确保Python环境正常；

2.  无需安装额外第三方依赖（基于Python原生库开发）；

3.  确保服务器端与客户端在同一网络环境下（本地部署时，默认IP：127.0.0.1）。

## 8.2 服务器端部署步骤

1.  将server.py文件复制到部署服务器（或本地电脑）；

2.  打开命令行，进入server.py所在目录；

3.  执行命令：python server.py；

4.  看到“服务器启动，监听端口8888”提示，说明服务器端部署成功，保持命令行窗口打开，不可关闭。

## 8.3 客户端部署步骤

1.  将client.py文件复制到用户电脑（或与服务器端同一台电脑）；

2.  打开命令行，进入client.py所在目录；

3.  执行命令：python client.py；

4.  客户端启动后，展示登录页面，说明客户端部署成功，可正常进行操作。

## 8.4 部署注意事项

1.  服务器端启动后，不可关闭命令行窗口，否则服务停止；

2.  客户端启动前，需确保服务器端已正常启动，否则无法连接；

3.  若端口8888被占用，可修改server.py和client.py中的端口号（确保两者一致）；

4.  服务器端重启后，内存中的预约数据会丢失，需提前告知用户。

# 9. 维护与调试说明

## 9.1 常见问题及解决方法

1.  客户端无法连接服务器：检查服务器端是否已启动，IP和端口是否一致，网络是否正常；

2.  预约失败：检查输入信息是否完整、格式是否正确，是否存在时间冲突；

3.  查询不到预约记录：检查预约ID、组织者姓名是否输入正确，是否存在该预约记录；

4.  取消预约失败：检查预约ID是否正确，该预约记录是否已被删除；

5.  程序崩溃：检查是否输入非法数据，是否存在代码逻辑bug，可通过print语句或调试工具排查。

## 9.2 调试方法

1.  服务器端调试：在关键代码行添加print语句，打印请求数据、响应数据、校验结果，排查逻辑问题；

2.  客户端调试：在按钮点击事件、RPC请求函数中添加print语句，打印输入数据、请求数据、响应数据，排查交互和请求问题；

3.  使用PyCharm调试工具：设置断点，逐步执行代码，查看变量值变化，定位bug位置。
