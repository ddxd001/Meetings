# 会议室预约管理系统产品需求文档（PRD）

# 1. 文档概述

## 1.1 项目背景

为解决办公楼会议室传统预约方式效率低下、易出现时间冲突、管理不规范等问题，基于Python语言开发会议室预约管理系统，采用RPC（远程过程调用）架构实现客户端与服务器端互联，完成题目要求的全部核心功能，实现办公楼会议室预约、查询、取消的线上化管理。该办公楼共6层，每层配置3个10人会议室和2个5人会议室，系统适配该配置场景，无需额外扩展功能。

## 1.2 项目目标

1.  采用Python语言开发，基于RPC架构实现客户端与服务器端互联，确保前后端通信稳定；

2.  严格实现题目要求的4个核心功能（预约会议室、按ID查询预约、按组织者查询预约、取消预约），无任何额外新增功能；

3.  保证数据存储可靠（服务器端内存存储）、业务逻辑正确（时间冲突校验等）；

4.  客户端提供简洁易用的交互界面，满足办公楼用户日常预约操作需求。

## 1.3 文档范围

本文档仅覆盖题目明确要求的功能模块、技术架构、数据结构、交互逻辑等内容，不包含权限管理、硬件联动、数据统计、审批流程等题目外扩展功能，聚焦核心需求的落地实现，为Python开发提供明确的需求依据。

## 1.4 开发语言与技术栈

核心开发语言：Python（纯Python实现，无额外第三方依赖，原生支持RPC相关通信）；

RPC实现：采用Python原生socket+多线程实现RPC通信（无需额外RPC框架，满足题目基础要求，适配核心功能）；

数据存储：服务器端采用内存列表存储预约数据（无需数据库，符合题目要求，重启后数据重置）；

客户端交互：Python GUI界面（采用tkinter库实现，模拟APP图形界面风格，支持点击操作、返回、登录、退出功能，简洁直观，聚焦核心操作，适配办公楼会议室预约场景）。

# 2. 系统架构设计

## 2.1 整体架构

系统采用客户端-服务器（C/S）架构，基于Python原生RPC机制实现前后端互联，核心分为客户端、服务器端两个模块，分工明确、通信高效，严格贴合题目要求，无多余架构设计。

### 2.1.1 客户端（Client）

核心职责：提供用户交互界面（模拟APP图形界面，采用tkinter库实现），支持点击操作，包含登录、核心功能入口、返回、退出功能，接收用户操作指令（预约、查询、取消），通过RPC调用服务器端对应接口，接收服务器端返回的结果，并在GUI界面清晰展示操作反馈。

### 2.1.2 服务器端（Server）

核心职责：启动RPC服务，监听客户端请求；负责预约数据的内存存储、业务逻辑处理（时间冲突校验、数据校验等）；对外暴露4个核心RPC接口，响应客户端的各类操作请求，返回处理结果。

### 2.1.3 RPC通信机制

采用Python原生socket实现远程过程调用，结合多线程处理客户端并发请求，完成数据的序列化与反序列化（JSON格式），屏蔽网络通信细节，实现客户端与服务器端的高效互联，确保操作指令与数据的准确传输。

## 2.2 架构约束

1.  仅采用Python语言开发，不使用其他编程语言；

2.  严格遵循C/S+RPC架构，不采用B/S架构；

3.  数据存储仅使用服务器端内存，不引入数据库、文件存储等额外存储方式；

4.  不添加任何题目外的扩展功能，仅实现核心4个功能；客户端采用模拟APP图形界面（tkinter库实现），支持点击操作，包含登录、返回、退出功能，不开发命令行界面。

# 3. 核心数据结构定义

系统核心数据结构为预约对象（Meeting），对应题目中的Appointment，字段严格按照题目要求定义，无额外字段，所有字段均为必填项，具体如下：

|字段名|字段类型（Python）|字段说明|约束条件|
|---|---|---|---|
|meetingId（预约ID）|int|预约记录的唯一标识，用于查询、取消预约|自增、全局唯一、不可重复，由服务器端自动生成|
|organizerName（组织者姓名）|str|会议发起人的姓名，用于按组织者查询预约|非空、不可为空字符串|
|roomName（会议室名称）|str|办公楼内被预约的会议室名称（结合办公楼配置：6层，每层3个10人会议室、2个5人会议室，命名格式：X层10人-1/2/3、X层5人-1/2，X为1-6）|非空、不可为空字符串，与办公楼实际会议室名称一致；办公楼共6层，每层设3个10人会议室（命名格式：X层10人-1、X层10人-2、X层10人-3，X为楼层数1-6）和2个5人会议室（命名格式：X层5人-1、X层5人-2，X为楼层数1-6），客户端输入需符合该命名规范。|
|topic（会议主题）|str|本次会议的主题或用途|非空、不可为空字符串|
|startTime（开始时间）|str|会议预约的开始时间，格式统一|非空，格式为“yyyy-MM-dd HH:mm”，早于结束时间，不可为过去的时间|
|endTime（结束时间）|str|会议预约的结束时间，格式统一|非空，格式为“yyyy-MM-dd HH:mm”，晚于开始时间|
|attendeeCount（参与人数）|int|本次会议的参会人数|非空、大于0的整数|
# 4. 核心功能需求

本系统仅实现题目要求的4个核心功能，每个功能严格按照题目描述开发，无额外扩展，具体需求如下：

## 4.1 功能1：预约会议室（对应接口：bookMeeting）

### 4.1.1 功能描述

用户通过客户端APP图形界面（模拟APP风格）完成登录后，在对应功能页面输入完整的预约信息（组织者姓名、会议室名称、会议主题、开始时间、结束时间、参与人数），客户端通过RPC调用服务器端bookMeeting接口，服务器端校验信息合法性及时间冲突后，完成预约数据存储，返回预约结果，客户端APP界面通过弹窗或指定区域展示结果。

### 4.1.2 输入要求

客户端需收集用户输入的全部预约字段，确保无空项，其中：

1.  开始时间、结束时间需严格遵循“yyyy-MM-dd HH:mm”格式；

2.  参与人数需输入大于0的整数；

3.  会议室名称需符合办公楼实际会议室命名规范（无非法字符），办公楼共6层，每层设3个10人会议室（命名格式：X层10人-1、X层10人-2、X层10人-3，X为1-6）和2个5人会议室（命名格式：X层5人-1、X层5人-2，X为1-6），输入需严格匹配该命名格式。

### 4.1.3 业务逻辑

1.  服务器端接收客户端传入的预约信息，首先校验所有字段的完整性和合法性；

2.  校验时间合法性：开始时间需早于结束时间，且开始时间不可为过去的时间（与服务器当前时间对比）；

3.  校验时间冲突：查询服务器端内存中该会议室的所有有效预约，若新预约时间段与已有预约时间段重叠（判定规则：新预约[s1, e1)与已有预约[s2, e2)满足e1 > s2 且 s1 < e2），则判定为冲突；

4.  校验通过后，服务器端为该预约分配唯一自增的meetingId，将预约信息存入内存列表；

5.  向客户端返回预约结果（成功/失败），若成功，同步返回生成的meetingId。

### 4.1.4 输出结果

客户端展示预约结果：

1.  预约成功：提示“预约成功，预约ID：XXX”；

2.  预约失败：提示“预约失败（时间冲突/输入格式错误/字段为空）”。

## 4.2 功能2：按ID查询预约（对应接口：queryByID）

### 4.2.1 功能描述

用户通过客户端APP图形界面（模拟APP风格）输入预约ID（整数类型），点击查询按钮，客户端通过RPC调用服务器端queryByID接口，服务器端根据预约ID查询内存中的预约记录，将查询结果返回给客户端，客户端APP界面清晰展示查询到的完整预约信息；若未找到对应ID，界面提示“无此预约ID，查询失败”。

### 4.2.2 输入要求

用户输入有效的预约ID（整数类型），不可为空、不可为非整数。

### 4.2.3 业务逻辑

1.  服务器端接收客户端传入的meetingId；

2.  遍历服务器端内存中的预约列表，匹配与meetingId一致的预约记录；

3.  若找到匹配记录，将该预约的完整信息（所有字段）返回给客户端；

4.  若未找到匹配记录，向客户端返回空结果。

### 4.2.4 输出结果

客户端展示查询结果：

1.  找到记录：以清晰格式展示该预约的所有字段信息；

2.  未找到记录：提示“无此预约ID，查询失败”。

## 4.3 功能3：按组织者查询预约（对应接口：queryByOrganizer）

### 4.3.1 功能描述

用户在客户端输入组织者姓名，客户端通过RPC调用服务器端queryByOrganizer接口，服务器端根据组织者姓名过滤内存中的预约记录，将该组织者发起的所有预约记录返回给客户端，由客户端展示。

### 4.3.2 输入要求

用户输入组织者姓名（字符串类型），不可为空，区分大小写（如“张三”与“张三 ”“张三”视为不同组织者）。

用户通过客户端APP图形界面（模拟APP风格）输入组织者姓名，点击查询按钮，客户端通过RPC调用服务器端queryByOrganizer接口，服务器端根据组织者姓名过滤内存中的预约记录，将所有匹配的预约列表返回给客户端，客户端APP界面以列表形式展示所有预约记录；若无匹配记录，界面提示“该组织者暂无预约记录”。

1.  服务器端接收客户端传入的organizerName；

2.  遍历服务器端内存中的预约列表，筛选出organizerName与输入一致的所有预约记录；

3.  无论是否找到匹配记录，均将筛选结果（列表形式）返回给客户端（无匹配记录则返回空列表）。

### 4.3.4 输出结果

客户端展示查询结果：

1.  有匹配记录：以列表形式展示每条预约的完整信息；

2.  无匹配记录：提示“该组织者暂无预约记录”。

## 4.4 功能4：取消预约（对应接口：cancelMeeting）

### 4.4.1 功能描述

用户在客户端输入预约ID（meetingId），客户端通过RPC调用服务器端cancelMeeting接口，服务器端根据预约ID查找对应的预约记录，找到则删除该记录，返回取消结果；未找到则返回取消失败。

### 4.4.2 输入要求

用户输入有效的预约ID（整数类型），不可为空、不可为非整数。

### 4.4.3 业务逻辑

1.  服务器端接收客户端传入的meetingId；

2.  遍历服务器端内存中的预约列表，匹配与meetingId一致的预约记录；

3.  若找到匹配记录，将该记录从内存列表中删除，向客户端返回“取消成功”；

用户通过客户端APP图形界面（模拟APP风格）输入要取消的预约ID，点击取消按钮，客户端通过RPC调用服务器端cancelMeeting接口，服务器端根据预约ID查找对应的预约记录，找到则删除该记录并返回“取消成功”，未找到则返回“取消失败”，客户端APP界面展示对应取消结果。

### 4.4.4 输出结果

客户端展示取消结果：

1.  取消成功：提示“取消成功”；

2.  取消失败：提示“取消失败，无此预约ID”。

# 5. 非功能需求

## 5.1 性能需求

1.  服务器端支持多客户端并发请求（至少支持5个客户端同时操作），无卡顿、无数据错乱；

2.  接口响应时间≤1秒，客户端操作反馈及时，无明显延迟；

3.  内存存储支持至少100条预约记录，满足办公楼日常使用需求。

## 5.2 可靠性需求

1.  服务器端启动后稳定运行，无异常崩溃；

2.  客户端与服务器端断开连接时，客户端给出明确提示，重新连接后可正常使用；

3.  输入非法数据时（如非整数的预约ID、格式错误的时间），系统不崩溃，给出明确的错误提示。

## 5.3 易用性需求

1.  客户端采用Python tkinter库开发，模拟APP图形界面，风格简洁，适配办公楼用户操作习惯，包含登录页面、核心功能操作页面，支持点击操作、返回操作、退出操作；

2.  所有操作反馈明确，成功/失败均通过APP界面弹窗或指定区域提示，便于用户快速知晓操作结果；

3.  输入控件适配APP界面风格（如输入框、按钮、列表框），操作便捷，支持异常输入校验，给出清晰提示。

2.  所有操作反馈明确，成功/失败均有对应提示，便于用户判断操作结果；

1.  客户端采用Python tkinter库开发GUI界面，界面简洁直观，包含4个核心功能的操作入口（预约、按ID查询、按组织者查询、取消），适配办公楼会议室配置；

## 5.4 兼容性需求

1.  支持Python 3.7及以上版本，无需额外安装第三方依赖；

2.  可在Windows、Linux、MacOS等主流操作系统上运行，客户端与服务器端可跨操作系统互联。

2.  进入任意功能页面后，界面顶部设有“返回”按钮，点击可返回主界面；

4.  操作完成后，界面展示操作结果，用户可点击“返回”按钮回到主界面，进行下一次操作。

3.  操作完成后，界面展示操作结果，用户可点击“返回”按钮回到主界面，进行下一次操作；

# 7. 数据需求

## 7.1 数据存储

1.  数据存储位置：服务器端内存（Python列表）；

2.  存储数据：仅存储预约对象（Meeting）的所有字段信息；

3.  数据生命周期：服务器端启动后，数据开始存储；服务器端关闭后，内存数据丢失，重启后重新初始化。

## 7.2 数据校验

1.  非空校验：所有预约字段不可为空，客户端输入时进行初步校验，服务器端再次校验；

2.  格式校验：时间字段严格遵循“yyyy-MM-dd HH:mm”格式，参与人数为大于0的整数；

3.  唯一性校验：meetingId全局唯一，由服务器端自动生成，确保无重复；

4.  逻辑校验：开始时间早于结束时间，不可预约过去的时间，避免时间冲突。

# 8. 风险与约束

## 8.1 风险说明

1.  服务器端重启后，内存中的预约数据会丢失，需提前告知用户；

2.  多客户端并发预约同一会议室同一时间段时，需确保时间冲突校验逻辑严谨，避免出现冲突预约；

3.  客户端与服务器端断开连接后，无法进行任何操作，需重新连接服务器。

## 8.2 约束条件

1.  开发语言必须为Python，不可使用其他编程语言；

2.  严格实现题目要求的4个核心功能，不添加任何额外功能；

3.  数据存储仅使用服务器端内存，不引入数据库、文件等额外存储方式；
