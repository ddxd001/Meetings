  阶段 2：重构目录结构

  把单文件代码拆成模块：

  meeting_room_system/
    server/
      app.py
      rpc_server.py
      service.py
      repository.py
      models.py
    client/
      app.py
      rpc_client.py
      pages.py

  建议职责：

  \- models.py：定义用户、会议室、预约数据结构
  \- repository.py：负责数据读写
  \- service.py：负责预约、查询、取消等业务逻辑
  \- rpc_server.py：负责 socket 通信
  \- rpc_client.py：负责客户端请求服务端
  \- pages.py：负责 Tkinter 页面
  \- app.py：启动入口

  阶段 4：增强业务规则

  按顺序加规则，不要一次全改：

1. 校验会议室是否存在。
2. 校验参会人数不能超过会议室容量。
3. 校验开始时间早于结束时间。
4. 校验不能预约过去时间。
5. 校验同一会议室同一时间段不能冲突。
6. 允许不同会议室同一时间段预约。
7. 允许同一会议室首尾相接预约，例如：


     10:00-11:00
     11:00-12:00
     这不算冲突。

  阶段 5：增加新功能

  建议按这个优先级：

1. 按时间段查询空闲会议室。
2. 按会议室查询预约记录。
3. 修改预约。
4. 用户登录。
5. 管理员查看全部预约。
6. 管理员管理会议室。


  如果时间有限，至少做前两个：

  按时间段查询空闲会议室
  按会议室查询预约记录

  这两个最贴合会议室预约系统。

  阶段 6：改进客户端界面

  把现在的手动输入改得更规范：

1. 会议室名称输入框改成下拉框。
2. 参会人数输入框只允许输入数字。
3. 查询结果用表格展示。
4. 登录后显示当前用户。
5. 增加“空闲会议室查询”页面。
6. 管理员和普通用户显示不同菜单。


  阶段 7：完善通信协议

  统一请求格式：

  {
    "action": "bookMeeting",
    "data": {
      "roomId": 1,
      "topic": "项目讨论",
      "startTime": "2026-06-02 10:00",
      "endTime": "2026-06-02 11:00",
      "attendeeCount": 6
    }
  }

  统一响应格式：

  {
    "success": true,
    "code": "OK",
    "msg": "预约成功",
    "data": {
      "meetingId": 1
    }
  }

  这样后续写文档和测试都更清楚。

  阶段 8：补测试

  至少写这些测试：

1. 预约成功。
2. 会议室不存在。
3. 人数超过容量。
4. 时间格式错误。
5. 开始时间晚于结束时间。
6. 预约过去时间。
7. 同会议室时间冲突。
8. 不同会议室同时间不冲突。
9. 首尾相接不冲突。
10. 取消预约成功。
11. 查询不存在的预约 ID。
12. 查询空闲会议室。
